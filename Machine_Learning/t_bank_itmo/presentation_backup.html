Бизнес-контекст

Т-Путешествия ежедневно обрабатывает тысячи предложений от поставщиков. Каждое предложение нужно сопоставить с мастер-комнатой в единой базе.

Проблема: если у поставщика недостаточно информации о номере — краудсорсинг не помогает, но деньги тратятся впустую.

Задача: заранее детектировать такие комнаты и не отправлять их в краудсорсинг.

Воркфлоу матчинга

📍 Определение hotel_id по адресу
↓
🤖 ML-модель: вероятность матча с мастер-комнатой
↓
✅ Уверенный матч → сохранить в БД
↓
❓ Нет матча → краудсорсинг (дорого!)

Постановка задачи

Что предсказываем

label = 1 — мало данных (не сматчена нужно задержать, не надо в крауд)

label = 0 — данных достаточно (комната сматчена -> в базу на разметку)

Ищем единички — комнаты с недостаточным контентом

Входные данные

  • supplier_room_name — название номера от поставщика
  • hotel_id — идентификатор отеля

Ключевые атрибуты номера

  • Уровень (Standard / Deluxe / …)
  • Ванная комната (своя / общая)
  • Конфигурация кроватей
  • Вместимость (кол-во человек)
  • Наличие балкона
  • Вид из окна
  • Семейный номер

Метрика

PR-AUC — полнота при высокой точности детекции критична для бизнеса

Выбор подхода

❌ TF-IDF + классик

Не учитывает семантику. Слова "twin" и "двуспальный" — разные токены.

✅ RuBERT + Hotel Emb

Понимает контекст и семантику. Учитывает специфику каждого отеля через эмбеддинги.

⚠️ LLM (GPT/LLaMA)

Избыточно. Медленно. Дорого. Задача не требует генерации.

Итоговый выбор: DeepPavlov/rubert-base-cased — русскоязычная BERT-модель, обученная на текстах на русском языке. Дообучается на нашей задаче (fine-tuning). Укладывается в ограничения: p95 < 15 сек, 1 GPU.

Почему hotel_id важен: в одном отеле "Standard" может означать одно, в другом — другое. Эмбеддинг отеля позволяет модели учитывать внутриотельную специфику.

Процесс обучения

Оптимизация

  • AdamW — стандарт для fine-tuning трансформеров
  • Linear warmup scheduler — первые 10% шагов LR растёт, затем линейно убывает
  • Mixed Precision (AMP) — ускорение на GPU, меньше памяти
  • GradScaler — предотвращает underflow градиентов при fp16

Сохранение модели

Сохраняется лучшая модель по PR-AUC на валидации. Каждый новый рекорд → best_model.pth

Цикл обучения

Forward pass (AMP autocast) - ускорение обучения x2, память GPU до 50%, точность не страдает
↓
BCEWithLogitsLoss - функция потерь бинарной классифкации (лучше для градиентов)
↓
scaler.scale(loss).backward() - обратное распространение градиентов с масштабированием
↓
scaler.step(optimizer) + scheduler.step() - защита от взрывов градиентов, обновляет веса
↓
Валидация → PR-AUC → сохранение лучшей

Ключевые технические решения

🏨 Hotel Embedding

Каждый отель получает обучаемый 16-мерный вектор. Модель учится, что в одном отеле "Standard" — это одно, в другом — другое.

Это критично: внутриотельная специфика существенна для матчинга.

🔒 view(-1) вместо squeeze()

При batch_size=1 squeeze() убирает лишние размерности и ломает форму тензора. view(-1) безопасен в любом случае.

⚡ Mixed Precision Training

torch.amp.autocast + GradScaler позволяют обучать быстрее и использовать меньше GPU-памяти без потери качества.

📝 Логирование

Все метрики пишутся в models/training_log.txt с временными метками. Удобно для мониторинга долгого обучения.

Почему PR-AUC, а не ROC-AUC?
PR-AUC фокусируется на детекции положительного класса. Для бизнеса важна высокая точность при высокой полноте — недовыдача комнат в краудсорсинг = прямые потери выручки.

Что делаем на каждой эпохе

  • Мониторим Train Loss (BCEWithLogitsLoss) по Val PR-AUC
  • Сохраняем лучшую модель

Интерпретация модели

На что опирается модель

Модель сильнее реагирует на отсутствие ключевых токенов:

  • Вид из окна: sea view, garden, mountain
  • Тип кровати: twin, double, king
  • Уровень: deluxe, superior, standard
  • Балкон: balcony, terrace
  • Ванная: ensuite, shared bath

Разница между hotel_id

Hotel Embedding обучается отдельно для каждого отеля:

  • Крупные отели с богатым ассортиментом — модель строже к полноте описания
  • Бутик-отели с 1-2 типами номеров — модель мягче
  • Новые поставщики (OOV) — используется усреднённый вектор

Вывод

Чем меньше специфических атрибутов в названии номера — тем выше вероятность label=1 (не отправлять в краудсорсинг).

  • «питание» −0.040 — информация о питании не входит в эталоны

💼 Бизнес-эффект

Снижение затрат на краудсорсинг. Гибкое масштабирование на новых поставщиков с низким качеством контента.

Ключевые факторы качества модели

🧠 Понимание языка

RuBERT понимает морфологию русского языка (падежи, суффиксы) — критично для названий номеров типа "двухместный", "семейный", "люкс".

Пример: модель понимает, что "двуспальная" и "double" — семантически близкие понятия

🏨 Контекст отеля

Эмбеддинг отеля позволяет модели учитывать специфику конкретного отеля. "Standard" в 5-звёздочном отеле ≠ "Standard" в хостеле.

🎯 Фильтрация шума

Модель научилась отличать "шумовые" слова (питание, количество комнат) от ключевых маркеров типа номера (deluxe, twin, balcony).

Игнорирует: "завтрак включён", "2 комнаты"
Фокусируется на: "superior", "sea view", "king bed"
