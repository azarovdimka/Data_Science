# 🏨 Детекция неполных описаний комнат для Т-Путешествия

## 📋 Описание задачи

### Контекст
Сервис **Т-Путешествия** ежедневно обрабатывает множество предложений по бронированию отелей от разных операторов (Booking.com, Expedia, Hotels.ru и др.). Для предоставления клиентам лучших цен необходимо сопоставлять эти предложения с единой сущностью — **мастер-комнатой**.

### Проблема
Часто поставщики предоставляют **неполные или неоднозначные описания комнат**, из-за чего:
- Невозможно однозначно идентифицировать комнату даже вручную
- Увеличивается число заданий в краудсорсинге
- Растут операционные издержки
- Снижается качество матчинга

### Цель
Научиться **заранее предсказывать**, что в описании комнаты недостаточно данных для матчинга, чтобы:
- Минимизировать поток таких заданий в краудсорсинг
- Запрашивать дополнительные данные у операторов
- Достичь заданной точности детекции (Precision ≥ 80%)
- Максимизировать полноту обнаружения (Recall)

---

## 🎯 Постановка ML-задачи

### Тип задачи
**Бинарная классификация**

### Целевая переменная
- `is_incomplete = 1` — описание неполное (недостаточно данных для матчинга)
- `is_incomplete = 0` — описание полное (достаточно данных)

### Метрики качества
1. **Precision (Точность)** — минимизация ложных срабатываний
   - Важно не отправлять в краудсорсинг полные описания
   
2. **Recall (Полнота)** — максимизация обнаружения неполных описаний
   - Важно выявить все проблемные случаи
   
3. **F1-Score** — баланс между Precision и Recall

4. **ROC-AUC** — общая способность модели различать классы

### Бизнес-требования
- **Precision ≥ 80%** — не более 20% ложных срабатываний
- **Максимальный Recall** при соблюдении требования по Precision

### Характеристики неполных описаний
- Много пропущенных значений (NaN)
- Пустые строки в текстовых полях
- Короткое описание или его отсутствие
- Мало удобств
- Отсутствие ключевых полей (тип кровати, вид, площадь)

## 🔧 Feature Engineering

### 1. Признаки на основе пропусков

```python
# Количество пропущенных важных полей
missing_count = количество NaN в [area, bed_type, view, max_guests, floor, n_rooms, rating]
```

**Логика**: Чем больше пропусков, тем выше вероятность неполного описания.

### 2. Признаки длины текстовых полей

```python
room_name_len = len(room_name)
description_len = len(description)
amenities_count = количество элементов в списке удобств
```

**Логика**: Короткие или пустые описания указывают на неполноту данных.

### 3. Бинарные признаки наличия данных

```python
has_area = 1 если area не NaN, иначе 0
has_bed_type = 1 если bed_type не пустая строка, иначе 0
has_view = 1 если view не пустая строка, иначе 0
has_max_guests = 1 если max_guests не NaN, иначе 0
has_floor = 1 если floor не NaN, иначе 0
has_n_rooms = 1 если n_rooms не NaN, иначе 0
has_rating = 1 если rating не NaN, иначе 0
has_description = 1 если description_len > 0, иначе 0
has_room_name = 1 если room_name_len > 0, иначе 0
```

**Логика**: Явное указание наличия/отсутствия каждого ключевого поля помогает модели выявлять паттерны.

### 4. Числовые признаки

```python
area, max_guests, floor, n_rooms, price, rating, n_photos
```

**Логика**: Косвенные индикаторы качества данных (например, низкий рейтинг может коррелировать с плохим качеством описаний).

## 🎯 Оптимизация порога классификации

### Проблема
По умолчанию модель использует порог 0.5 для классификации:
- `probability >= 0.5` → класс 1 (неполное)
- `probability < 0.5` → класс 0 (полное)

Но это не всегда оптимально для бизнес-задачи!

### Решение
Построение **Precision-Recall кривой** и выбор порога, который:
1. Обеспечивает **Precision ≥ 80%** (бизнес-требование)
2. Максимизирует **Recall** при этом ограничении

### Алгоритм

```python
1. Получить вероятности для тестовой выборки
2. Построить Precision-Recall кривую для разных порогов
3. Найти все пороги, где Precision ≥ 0.80
4. Выбрать порог с максимальным Recall
5. Использовать этот порог для финальных предсказаний
```

### Пример результата
```
Оптимальный порог: 0.35
Precision: 0.82
Recall: 0.91
F1-Score: 0.86
```

**Интерпретация:**
- 82% предсказаний "неполное" действительно неполные (мало ложных срабатываний)
- 91% всех неполных описаний обнаружены (высокая полнота)

## 💡 Бизнес-логика решения

### Workflow внедрения

```
1. Оператор отправляет описание комнаты
   ↓
2. Система извлекает признаки
   ↓
3. ML-модель предсказывает вероятность неполноты
   ↓
4. Если вероятность > порог (0.35):
   ├─→ Отправить запрос оператору на дополнение данных
   └─→ Указать, какие поля отсутствуют
   
5. Если вероятность ≤ порог:
   └─→ Отправить на матчинг с мастер-комнатой
```

### Преимущества подхода

1. **Проактивность** — выявление проблем до краудсорсинга
2. **Экономия** — снижение потока в краудсорсинг на 20-30%
3. **Качество** — улучшение данных у источника
4. **Автоматизация** — минимум ручной работы

    model = pickle.load(f)

    scaler = pickle.load(f)

# Предсказание
def predict_incomplete(room_data):
    # room_data - DataFrame с признаками
    X = create_features(room_data)  # Создание признаков
    X_scaled = scaler.transform(X)  # Масштабирование
